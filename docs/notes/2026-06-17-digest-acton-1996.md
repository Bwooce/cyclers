## 2. What the paper actually is

The canonical **design-level overview** of NASA's SPICE ancillary-information system, written by its lead architect at JPL/NAIF. It is a *system description paper*, not an algorithm paper: there are no equations, no tables of numerical data, no benchmark results. The paper introduces SPICE to the planetary-science community (the journal's audience), lays out the kernel-file architecture, the NAIF Toolkit / SPICELIB API model, the producer/consumer responsibility split, the portability strategy, and the customer-support / distribution policy. It also names the missions that had adopted SPICE by Spring 1995 (Table 2) and the platforms NAIF supports (Table 1).

The paper traces SPICE's origin to two documents: the CODMAC 1982 report (NASA's space-science data-handling assessment, which identified ancillary-data production and distribution as a gap) and the November 1983 NASA Planetary Data Workshop. SPICE was the subsequent specification effort and was being applied across NASA's planetary program by the time of writing (Galileo, Mars Global Surveyor, Mars Pathfinder, NEAR, Cassini already in flight or in development). The paper closes with a directory entry for ordering SPICE products from NAIF, free of charge, by e-mail to `cacton@naif.jpl.nasa.gov` (the author's address).

It is the foundational design document for the SPICE system. By 1996 SPICE was 13 years into its life; it remained recognisably the same system into the 2020s, so this paper has held up as a faithful description of the architecture the project actually uses today (DE440 BSP, naif0012.tls LSK, URA111 satellite BSP, spiceypy bindings).

## 3. SPICE design overview — kernel file types and toolkit pattern

### 3.1 Kernel file taxonomy (p. 66 Fig. 1)

Acton's Fig. 1 lays out the SPICE Data Products as a one-to-one mapping from logical element to file type:

| Logical element | Kernel | Contents (Fig. 1 right-hand column, p. 66) |
|---|---|---|
| **S** (Spacecraft) | **SPK** | "Position and velocity of spacecraft and target bodies, given as a function of time" |
| **P** (Planet) | **PCK** | "Selected target body physical and cartographic constants" |
| **I** (Instrument) | **IK** | "Instrument information such as mounting alignment, field-of-view specifications, internal timing and command dictionary" |
| **C** (C-matrix) | **CK** | "Inertially referenced attitude (pointing) for a spacecraft instrument platform, given as a function of time" |
| **E** (Events) | **EK** | "A combination of science plan, spacecraft and instrument command logs, and mission operations electronic notebooks" |
| (none) | **LSK, SCLK** | "Tabulations of leapseconds and spacecraft clock coefficients needed for time conversions" |
| **S** (Software) | **SOFTWARE** | "A combination of the NAIF Software Toolkit and a user's own software" |

The acronym SPICE is the first letter of each *logical element* in Fig. 1: S(pacecraft), P(lanet), I(nstrument), C(-matrix), E(vents). Acton notes (p. 66 col. 1) "the acronym might better have been 'SPICES', with the final 'S' standing for software." LSK / SCLK do not have a letter in the acronym; FK (frames kernel) is not mentioned in this paper — it appears in the literature later but is absent from Acton 1996's Fig. 1 (NB: this is sometimes assumed by modern users to have been there from the start; it was not).

### 3.2 Producer/consumer split (p. 66 col. 2)

A specific design choice spelled out by Acton: "Instances of each type of SPICE kernel file should be produced by the entity most qualified to do so — the person or group most familiar with the components and processes from which ancillary data are produced." Examples given:
- **SPK files** produced by a project's **navigation team** (they estimate the orbit).
- **CK files** produced by the **spacecraft control team** (they own attitude).
- NAIF *does* produce **SPK and PCK files from generic data** not tied to a flight project — this is the lineage of today's DE440 BSP, the URA111 satellite kernel, the naif0012.tls LSK that the project consumes. NAIF is the producer of these generics; NAIF is **not** the producer of mission-specific SPKs.

### 3.3 Distribution model (p. 66 col. 2)

- During the mission: kernels live at the mission operations center.
- At mission end: kernels transfer to the **NAIF node of PDS** for formal archiving.
- Customer orders are filled from this PDS archive.
- NAIF Toolkit software is distributed by NAIF directly. Delivery via Internet (most common in 1995), CD-R, magnetic tape, or diskette.

### 3.4 NAIF Toolkit / SPICELIB (p. 66-67)

- **SPICELIB** = a library of ANSI FORTRAN 77 subroutines and functions.
- Users *do not need* to understand kernel file structure — the SPICELIB reader subroutines abstract it. Example given: `SPKEZ` returns target position relative to observer at user-specified epoch.
- SPICELIB also includes time translation, reference frame and coordinate conversion, lighting-angle computations, solid geometry, vector and matrix algebra, conic-element applications, string/parsing/array utilities.
- The Toolkit also ships **"cookbook programs"** — sample programs demonstrating SPICELIB use — and **utility programs** for conversion / summarization / labeling of binary kernel files.
- Flight projects may extend SPICELIB with mission-specific subroutines (example given: `GLLSPICE` for Galileo).
- C-language re-coding was in progress at time of writing (p. 68 last paragraph of "Portable data and software"). This eventually became **CSPICE**, which is what spiceypy binds against today. In 1996 it did not yet exist.

### 3.5 Two usage patterns (p. 67-68 Figs. 2-3)

- **Fig. 2 (p. 67):** "Full" pattern. User's application program integrates SPICELIB calls with user subroutines; SPK + PCK + IK + CK + EK + Misc kernels feed in; instrument data also feeds in; user produces both science/engineering results AND optional **"Kernel Updates"** that feed back into the SPICE archive (with peer review).
- **Fig. 3 (p. 68):** Two-step "lite" pattern. User's Geometry Program reads SPK + PCK + LSK only, producing Derived Geometry Data. The Derived Geometry Data + Instrument Data + Calibration Data then feed a separate User's Analysis Program that produces Scientific Results. The point Acton makes (p. 67 col. 2): "For many applications a user needs data from just a few SPICE files."

### 3.6 Characteristics design choices

- **Portability (p. 68 col. 2):** Text-format kernels port directly. Binary kernels port via **SPICE Transfer Format** (encoded text, ~2.5× the binary size) — a portable NAIF utility converts binary↔text. SPICELIB uses ANSI FORTRAN 77 with platform dependencies "minimized, isolated and clearly documented". Table 1 (p. 69) lists 10 platforms NAIF supports plus 5 customer-ported platforms.
- **Extensibility (p. 69 col. 1):** Two mechanisms.
  1. Text kernels use a `KEYWORD = VALUE` style, allowing new keyword/value pairs to be added without breaking old readers.
  2. Binary kernels are built on data structures supporting **a family of data types, unlimited in number**. Acton notes the SPK family **"started with just three data types, currently consists of 16 data types, such as Chebyshev polynomials, traditional conic elements and NORAD's two-line elements"** (p. 69 col. 1). This is the same SPK Type architecture (Type 2, Type 9, Type 10, Type 13, Type 18, Type 21, …) that NAIF still ships.
- **Code maintainability (p. 69 col. 1):** "Approximately 70 percent of the source 'code' is actually documentation." NAIF avoids FORTRAN features that hurt maintainability (`COMMON` blocks, `EQUIVALENCE` statements).

### 3.7 Applications (p. 69-70 Table 2)

Spring 1995 status of flight project SPICE usage:
- **Completed missions (incl. data restorations):** Viking, Pioneer 10/11, Voyager, Magellan, Phobos 88, Ulysses, Clementine.
- **Current applications:** Galileo, Mars Global Surveyor, Mars Pathfinder + Mars 98, Mars 96, NEAR, Radioastron/VSOP, Cassini.
- **Future possibilities:** Eos, Discovery missions, MSTI 3, Pluto Fast Flyby, Solar Probe, SIRTF, New Millennium missions, Rosetta, Japanese Lunar/Mars missions.
- Informal uses (p. 70): Russian Granat, Interbol, Spectrum-X/G; terrestrial observatory planning; mission concept studies; solar system dynamics analyses.

## 4. Project relevance — does our SPICE usage follow Acton 1996's patterns?

**YES. The project follows Acton 1996's documented standard usage patterns. No gaps to fix.**

Specific check points:

### 4.1 Producer/consumer split — we are pure consumers of generic kernels

- `src/cyclerfinder/verify/spice_kernels.py` (the V4-strict / cross-check infrastructure) and `src/cyclerfinder/data/validation/v4_uranus_strict.py` (Uranian V4-strict) both consume **generic NAIF-produced kernels**: DE440 BSP, naif0012.tls LSK, URA111 BSP. These are exactly the "generic SPK and PCK files" Acton (p. 66 col. 2) describes NAIF as producing. The project produces no kernels of its own. This matches the design: we are a consumer-only application, the analogue of Acton's Fig. 3 "User's Geometry Program" two-step pattern.

### 4.2 SPICELIB / CSPICE API usage — standard patterns

Inventory in `v4_uranus_strict.py`:
- `spice.furnsh(p)` for each kernel path (LSK + PCK + URA111 BSP), with `spice.kclear()` in a `try/finally` to avoid kernel-pool contamination across runs.
- `spice.str2et(launch_epoch_utc)` for UTC→ET conversion.
- `spice.spkezr(target, et_seconds, "J2000", "NONE", observer)` for state retrieval.

These are exactly the SPICELIB reader-subroutine patterns Acton describes (p. 66-67 col. 1-2: "the interface to these datasets is always realized through the SPICE kernel file 'reader' subroutines found in SPICELIB. For example, calling SPICELIB routine SPKEZ returns the position of a target relative to an observer at a user-specified epoch."). `SPKEZ` is the FORTRAN 77 name; `spkezr` is its CSPICE/spiceypy equivalent with named-body string input. **The project pattern is textbook Acton 1996.**

### 4.3 Same-kernel cross-check pattern (Acton-aligned)

`tests/verify/test_ephemeris_crosscheck.py` (#129 ephemeris cross-check) deliberately points NAIF's CSPICE reader (spiceypy) at the **same DE440 BSP astropy caches** and compares state vectors. This is Acton-aligned by design: the SPICE kernel is an *independent specification* of state data, and a same-kernel-different-reader comparison isolates frame-convention / time-scale / reader-implementation bugs. The strategy is consistent with Acton's "Correct results are obtained" section (p. 67-68), which lists the prerequisites for correctness (correct data, correct software, correct labels, correct subroutine choice) — our test pins all four.

### 4.4 "Seeds, not tracks" — Acton's design is acknowledged contrast, not adoption

`docs/spec.md` lines 1140-1153 already cites Acton 1996 explicitly as the **design point we deliberately *do not* adopt**: SPICE SPK/DAF stores *sampled trajectories* (Chebyshev-fitted state histories — confirmed by Acton p. 69 col. 1 listing Chebyshev polynomials as an SPK data type), but the catalogue stores *generative seeds* (elements, invariants, encounter sequence) and regenerates trajectories on demand. The project's spec.md provenance note (line 1132-1138) said Acton 1996 "remains cited from its bibliographic record only (paywalled)" — that note is now stale: we have actually read the paper, and the citation IS faithful to the source. Recommended spec.md amendment in §7.

### 4.5 LSK / time-system handling

Acton mentions LSK only as "tabulations of leapseconds … needed for time conversions" (p. 66 Fig. 1). The project loads `naif0012.tls` via `spice.furnsh` and uses `spice.str2et` for UTC→ET conversion (v4_uranus_strict.py line 255) — this is the standard LSK consumption pattern. Project does NOT use SCLK (no spacecraft clock — the catalogue is mission-agnostic). LSK-only is the correct subset for our use case (ephemeris-only consumer, no telemetry).

### 4.6 Possible SPK data types we could exploit but currently don't

Acton p. 69 col. 1 says SPK kernels are an **extensible family of data types** — by Spring 1995 there were already 16 (Chebyshev polynomials, conic elements, NORAD two-line elements). The project today reads only SPK Type 2 / Type 3 (Chebyshev position + velocity, used by DE440) and whatever Type is in URA111. The project does not produce SPK kernels, so producing in any Type is N/A. As a consumer the project does not care about Type — `spkezr` is type-agnostic. **No gap.** (For reference, SPK Types added after Acton 1996 include Type 21 — extended difference lines, Type 18 — ESOC weighted Hermite, and others; spiceypy handles all of them transparently.)
