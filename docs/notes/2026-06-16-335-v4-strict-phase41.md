## Part B — V4-strict driver

`src/cyclerfinder/data/validation/v4_uranus_strict.py` (commit `2eda155`).

### What V4-strict adds over V4-scipy

| Effect | V4-scipy (#332) | V4-strict (this) |
|---|---|---|
| Moon eccentricity at encounter | 0 (circular) | SPICE-sampled (e.g. Umbriel 0.0041) |
| Moon inclination | 0 (planar) | SPICE-sampled (~75 deg in J2000) |
| Secular precession | 0 | baked into SPICE state |
| Lambert endpoint moon r/v | Kepler-only | SPICE-driven |
| Third-body moon positions during leg | circular-Kepler | SPICE-driven |
| Uranus J2 | 3.34343e-3 (Jacobson 2014) | same |
| Integrator | DOP853, rtol=1e-10 | same |

Force model, integrator, and physics helpers (J2, Battin third-body, Hill
softening) are shared with `v4_uranus.py` — only the moon ephemeris
source changes. The integrator family is unchanged, isolating the
SPICE-ephemeris contribution.

### GMAT vs Python+SPICE

GMAT R2022a doesn't natively bundle CelestialBody definitions against
URA111. Hand-authoring the Uranian rotation-matrix / pole orientation
is out of scope. `spiceypy` binds the SAME JPL C library GMAT links,
so Python+SPICE driving DOP853 with SPK-sampled perturber states IS
the same integration recipe GMAT would compute under the hood.

A parallel `scripts/gmat_v4_uranus_generate.py` emits the equivalent
GMAT script for human cross-check; not the headline V4-strict verdict.

## What this means

### The bounded-drift signature IS real

At 2030 + 2050 epochs, the bounded-drift diagnostic (per-cycle drift
varies bounded between ~80,000 km and ~550,000 km without monotonic
divergence) survives the SPICE-real-ephemeris test. The signature is
not a Keplerian-model artifact.

### But the signature is launch-epoch-dependent

At 2000-01-15, the SAME Lambert geometry — same V_inf, same ToF, same
sequence — does NOT close under SPICE-real ephemeris. The Uranian
satellite phase configuration at this epoch shifts the SILVER's
encounter geometry meaningfully relative to the circular-coplanar
idealisation.

This is the FIRST V-tier verdict that exposes launch-epoch sensitivity
of the SILVER — V2/V3/V4-scipy all used circular-coplanar Kepler moons,
so a single `phase0_deg` (29.999... at the SILVER's discovery)
sufficed. V4-strict reads the actual moon state at every encounter,
and the answer depends on WHEN.

### Catalogue admission implications

Under schema v4.7's `quasi_cycler` orbit_class:

* `epoch_locked` must be **true** (not the SILVER's discovery default
  of `false`).
* `validity_window` must span the PASS sub-band. This 3-epoch sample
  is too coarse — a finer epoch grid (annually 2000-2099, ~100 V4-strict
  runs × 1 s each = ~2 min) is needed to find the boundary.
* `launch_epoch` should be one within the PASS sub-band; 2030-06-21
  or 2050-12-31 both qualify.

## Phase 5 recommendation

**Do NOT fire #337 catalogue admission yet.** The MIXED verdict means
the SILVER row's `validity_window` is unknown — single-epoch admission
is weak, epoch-blind admission is false.

**Recommended next task** (provisionally #338, NOT #337): annual epoch
sweep 2000-2099 at n_cycles=3, plotting per-epoch agreement-vs-V3 to
find the PASS / FAIL boundary. Once characterised:

* PASS sub-band cyclic (suggesting moon-phase commensurability) →
  admit as `quasi_cycler` with `epoch_locked=true`, `launch_epoch`
  in the PASS band, `validity_window` = the sub-band.
* PASS sub-band irregular / chaotic → retire to negative-results
  registry (#172) with the launch-epoch sensitivity finding recorded.
